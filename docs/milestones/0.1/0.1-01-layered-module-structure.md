# 0.1-01 — Слоистая структура модулей

**Milestone:** 0.1 — Foundation and project skeleton  
**Тип:** структура / рефакторинг каркаса

## Цель

Создать каталоги и точки входа слоёв согласно `docs/TECHNICAL_DESIGN.md` §4, чтобы дальнейший код размещался по правилу зависимостей `UI → Application → Domain`, а Infrastructure подключалась только через composition/DI.

## Зависимости

Нет (стартовая задача milestone 0.1).

## Объём работ

- Создать дерево каталогов под `src/` (минимум):
  - `composition/`
  - `application/` (`services/`, `workflows/`, `ports/`)
  - `domain/` (`track/`, `place/`, `links/`, `shared/`)
  - `infrastructure/` (`storage/`, `parsers/`, `obsidian/`, `logging/`, `map/`)
  - `ui/` (`views/`, `settings/`, `components/`, `i18n/`)
- Добавить barrel/index или placeholder-модули там, где нужны явные границы импорта (пустые экспорты допустимы).
- Перенести или согласовать существующий прототипный код (например `src/views/`, `src/map/`) в целевые слои **без** изменения поведения — допустим временный re-export из старых путей, если это нужно для непрерывной сборки.
- Обновить пути импортов в `esbuild`/tsconfig при необходимости.

## Вне scope

- Реализация бизнес-логики, storage, парсеров.
- Удаление legacy settings/commands (задача **0.1-13**).

## Условия завершения

- [ ] В репозитории присутствуют все перечисленные корневые слои и подкаталоги из TECHNICAL_DESIGN §4 (допускаются пустые `index.ts` / placeholder-файлы).
- [ ] `npm run build` завершается успешно после реорганизации путей.
- [ ] В `src/main.ts` (или эквивалентной точке входа) нет новой feature-логики — только lifecycle и делегирование (подготовка к **0.1-07**).
- [ ] В README milestone или комментарии в PR кратко зафиксировано, какие legacy-пути (`src/views`, `src/map` и т.д.) ещё остаются и будут перенесены позже (если re-export оставлен).

## Артефакты

- Структура каталогов `src/{composition,application,domain,infrastructure,ui}/...`
