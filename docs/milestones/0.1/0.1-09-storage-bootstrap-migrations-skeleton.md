# 0.1-09 — Bootstrap storage-адаптера и каркас миграций

**Milestone:** 0.1 — Foundation and project skeleton  
**Тип:** infrastructure

## Цель

Подключить **выбранный** в **0.1-05** storage adapter в composition: инициализация при старте плагина, каркас `migrations.ts` (версионирование, runner, без полной v1-схемы).

## Зависимости

- **0.1-05** — [Зафиксировать решение по storage](./0.1-05-record-storage-decision.md)
- **0.1-07** — [DI-контейнер и bootstrap](./0.1-07-di-container-bootstrap.md)

Рекомендуется после **0.1-08** (замена stub repositories), но минимально достаточно **0.1-07** + **0.1-05**.

## Объём работ

- `infrastructure/storage/storage-adapter.ts` — фасад открытия/закрытия БД в plugin data dir.
- `infrastructure/storage/migrations.ts`:
  - таблица/ключ `schema_version` (или эквивалент в meta),
  - `runMigrations(currentVersion)` с пустой цепочкой или migration `v0 → v0` placeholder,
  - логирование start/end/error через logger port.
- В `onload` контейнера: open → migrate → register repository implementations (минимальный meta-repository допустим).
- Обработка ошибок: Notice + запись в logger, плагин не падает без catch.

## Вне scope

- Полная логическая схема v1, индексы, CRUD треков (**0.2**).
- Rotating file logs (**0.2**).

## Условия завершения

- [ ] При первом запуске создаётся файл/хранилище в plugin data directory (путь соответствует **0.1-05**).
- [ ] Повторный запуск не дублирует инициализацию (идемпотентный bootstrap).
- [ ] Migration runner вызывается на каждом `onload`; при отсутствии миграций — no-op успешно.
- [ ] Concrete storage package импортируется **только** из `infrastructure/storage/**` (architecture gate).
- [ ] `npm run build` проходит; desktop smoke: enable plugin → disable → enable, данные сохраняются.

## Артефакты

- `src/infrastructure/storage/storage-adapter.ts`
- `src/infrastructure/storage/migrations.ts`
- Регистрация в `composition/container.ts`
