# 0.1-04 — Spike: feasibility парсера FIT / FIT.GZ

**Milestone:** 0.1 — Foundation and project skeleton  
**Тип:** исследование / gate

## Цель

Выбрать кандидата библиотеки/пути реализации парсера `.fit` и `.fit.gz`, проверить совместимость с esbuild, влияние на размер бандла и runtime desktop/mobile, распарсить репрезентативные фикстуры в промежуточную модель.

## Зависимости

Нет блокирующих зависимостей от **0.1-01** / **0.1-02** (можно параллельно).

Желательно согласовать форму **промежуточной модели** с **0.1-02** (parser port); при расхождении — обновить порт в **0.1-02** или зафиксировать delta в отчёте spike.

## Объём работ

- Подобрать 1–2 кандидата (npm или thin wrapper) для FIT binary.
- Реализовать spike-парс `.fit` и `.fit.gz` (gzip decompress → parse) на тестовых фикстурах в `tests/fixtures/` или временной папке.
- Измерить: размер добавки к `main.js`, время cold parse на 1–2 файлах, ошибки в Obsidian desktop/mobile sandbox.
- Сверить выход с ожидаемыми полями промежуточной модели (треки/сегменты, timestamps, опционально HR/cadence/power — как в REQUIREMENTS).
- Если feasibility **не проходит** — описать альтернативы: отложить FIT до post-v1, desktop-only parser path, и т.д.

## Вне scope

- Production parser router и нормализация метрик (**0.4**).
- Запись финального решения в TECHNICAL_DESIGN (**0.1-06**).

## Условия завершения

- [ ] Есть минимум **две** тестовые фикстуры: один `.fit`, один `.fit.gz` (или обоснование одной, если вторая недоступна).
- [ ] Для каждого кандидата задокументированы: bundle impact, desktop result, mobile result (или «не проверено» с планом).
- [ ] Пример выхода spike (JSON/таблица полей) приложен к отчёту.
- [ ] Рекомендация: **go** (библиотека X) / **go with constraints** / **no-go** для v1.
- [ ] Spike не ломает `npm run build` production-сборки (изолированный модуль или dev-only script).
