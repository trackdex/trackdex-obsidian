# Trackdex — требования к продукту (v1.0)

Документ фиксирует продуктовые и функциональные требования к плагину **Trackdex** для Obsidian. Собран по итогам продуктового интервью; дополняет [CONCEPT.md](./CONCEPT.md).

**Статус:** утверждён для v1; `REQUIREMENTS.md` является основным источником правды, `CONCEPT.md` может содержать устаревшие решения.  
**Целевая публикация:** [Obsidian Community Plugins](https://obsidian.md/plugins) в обозримом будущем.

---

## 1. Видение

**Trackdex** — Obsidian-плагин для **офлайн-каталогизации GPS-треков**: просмотр, систематизация, статистика, привязка к именованным местам.

| Аспект | Решение |
|--------|---------|
| Позиционирование | Библиотека / индекс треков, не GIS-редактор и не замена Strava |
| Аудитория v1 | Велосипедисты |
| Аудитория далее | Бег, походы, любой sport/outdoor с GPS-треками |
| Редактирование треков | Не в ближайших версиях (read-only) |
| Синхронизация Strava/Garmin | v2+ |
| Платформа | Desktop + mobile; стремиться к одинаковому поведению, mobile-риски фиксировать отдельно |
| Язык UI | EN + RU; следует языку Obsidian |

---

## 2. Принципы

1. **Zero-config** — пользователь открывает существующий vault с GPX/TCX/FIT/FIT.GZ; плагин работает без переноса файлов и без обязательной структуры каталогов.
2. **Треки где угодно** — скан всего vault; пользователь сам раскладывает коллекцию.
3. **Read-only** — файлы `.gpx` / `.tcx` **не модифицируются**.
4. **Obsidian-native** — связи через `[[wikilink]]`; места (places) — обычные заметки; неоднозначность ссылок — как в Obsidian.
5. **Предрасчёт** — метрики и упрощённая геометрия в локальном служебном индексе; полный парсинг не при каждом открытии.
6. **Контроль индексации** — первый scan только после явного согласия; после этого фон по умолчанию, прогресс и статусы всегда видны; пользователь может управлять процессом.

---

## 3. Пользовательские сценарии (v1)

1. Пользователь открывает vault с сотнями GPS-треков (GPX/TCX/FIT/FIT.GZ) в произвольных папках → видит объяснение, что индекс ещё не создан, и запускает первый scan явной кнопкой.
2. После первого согласия plugin выполняет фоновые scan/инкрементальную индексацию с индикатором прогресса; если индексация была прервана отключением plugin, при следующем включении пользователь видит «индексация была прервана» и кнопку «проверить и продолжить».
3. Клик по `.gpx` / `.tcx` / `.fit` / `.fit.gz` в файловом менеджере → view: карта + статистика.
4. Sidebar «Треки»: группировка по дате (год → месяц → день), по place, по виду спорта; сортировка и фильтры.
5. Создаёт place-заметку или преобразует текущую заметку в place через команду → треки привязываются по геометрии.
6. В журнальной заметке пишет ссылку на трек любым валидным для Obsidian способом (`[[20240505.gpx]]`, markdown-link, alias/embed/path variants) → в UI трека отображаются связанные заметки.
7. Смотрит км и часы за месяц с фильтром по виду спорта; сортирует places по дате последнего визита.

## 4. Функциональные требования

### 4.1. Индекс и скан vault

| ID | Требование | Приоритет |
|----|------------|-----------|
| F-01 | Рекурсивный scan `.gpx`, `.tcx`, `.fit`, `.fit.gz` **по всему vault**; matching расширений case-insensitive на Windows/Linux/macOS | Must |
| F-01b | Exclude по умолчанию: `.obsidian/`, `.trash/`; настраиваемые exclude-patterns в settings | Must |
| F-01c | Zero-config при первом включении плагина, но первый scan начинается только после явного согласия пользователя | Must |
| F-01d | Полный фоновый скан + инкремент на события vault (`create`, `modify`, `delete`, `rename`) | Must |
| F-01e | Throttling / батчи — не блокировать UI; умеренное потребление CPU и памяти | Must |
| F-01f | Визуализация индексации: ненавязчивая панель прогресса внизу справа; для больших файлов допустим статус `processing large file` без точного процента внутри файла | Must |
| F-01g | Статусы файла: `pending`, `indexing`, `indexed`, `stale`, `error` | Must |
| F-01h | Непроиндексированные, ошибочные и успешно обработанные файлы с отсутствующими полями явно помечены в UI; пользователь может скрыть необработанные треки | Must |
| F-01i | Управление: «Сканировать», пауза автоскана, «Переиндексировать places», «Пересчитать всё с нуля» | Must |
| F-01m | Если plugin был отключён во время индексации, при следующем включении не продолжать молча: показать состояние «индексация была прервана» и кнопку «проверить и продолжить» | Must |
| F-01n | «Пересчитать всё с нуля» удаляет только служебный индекс Trackdex и требует подтверждения с текстом, что файлы треков и заметки не меняются | Must |
| F-06 | Детект дубликатов по SHA-256 | Backlog |
| F-07 | Форматы v1: GPX + TCX + FIT + FIT.GZ | Must |

После первого согласия пользователя на индексацию автоскан включён при следующих запусках, пока пользователь не поставил паузу.

Удаление трека из vault удаляет его из каталога и индекса после обработки события `delete`; история удалённых/переименованных файлов не хранится.

Пользовательские exclude-patterns должны быть привычны пользователям Obsidian: glob/ignore-like patterns по vault-relative paths.

**Мульти-сегментные файлы** (несколько `<trk>` / `<Track>` в одном файле):

| ID | Требование |
|----|------------|
| F-01j | Один файл = **одна запись** в каталоге; метрики агрегируются по всем сегментам |
| F-01k | В UI просмотра — **список сегментов** внутри файла (если их несколько) |
| F-01l | Детали агрегации уточняются на реальных multi-track файлах при реализации |

### 4.2. Метрики

| ID | Требование | Приоритет |
|----|------------|-----------|
| F-33 | **Дата трека** = timestamp первой точки GPX / StartTime TCX | Must |
| F-05a | Агрегаты: км и часы за месяц / год / произвольный диапазон | Must |
| F-05a2 | Фильтр агрегатов по **виду спорта** | Must |
| F-05c | Средняя и максимальная скорость по треку | Must |
| F-05e | Набор и сброс высоты; HR, cadence, power — если присутствуют в файле | Must |
| F-05b | Графики активности по времени | v2+ |

**Алгоритмы (зафиксировано):**

| Метрика | v1 | v1.1+ |
|---------|-----|-------|
| **Elapsed time** | Разница между timestamp последней и первой точки | — |
| **Moving time** | Не показывается | Отсечение остановок по порогу скорости/времени |
| **Средняя скорость** | `distance / elapsed` | `distance / moving time` при наличии moving time |
| **Elevation gain** | См. [§ 4.2.1](#421-elevation-gain) | — |

**Источники данных и отсутствующие поля:**

1. Для общей статистики использовать значения из файла, если формат предоставляет готовые summary/metric values; если таких значений нет — пересчитывать из точек.
2. В UI просмотра трека желательно показывать и значения из файла, и пересчитанные значения (например, отдельными вкладками). Если это усложняет v1, допускается один основной источник с расширением в следующих версиях.
3. Не придумывать fallback-значения для отсутствующих данных. Если нет даты, координат, высоты, HR, power, sport и т.п. — честно показывать, что данных нет.
4. Статус `partial` не вводится. Валидный файл с неполным набором данных имеет статус `indexed`, а доступность конкретных полей определяется расширяемыми field/data flags (`has_geometry`, `has_time`, `has_elevation`, `has_hr`, `has_power`, `has_sport`, `has_file_metrics` и будущие поля).
5. Треки без даты группируются в отдельную группу вроде «Дата не указана». Треки без координат остаются в каталоге, но карта и привязка к places недоступны.
6. Дистанция v1 пересчитывается как сумма расстояний между соседними точками без фильтра GPS-скачков. Фильтры/сглаживание — будущая опция.

**Время и timezone:**

1. Пользователь видит время в своей текущей локальной timezone.
2. Для timestamp с явным UTC/offset хранить исходное значение и нормализованное UTC-время; при смене timezone пользователя UI отображает время в новой локальной timezone.
3. Для timestamp без timezone/offset трактовать время как локальное время пользователя на момент индексации; хранить исходную строку, рассчитанное UTC-время и timezone/offset индексации.
4. UI должен показывать timezone/offset, чтобы пользователь понимал источник времени.

#### 4.2.1. Elevation gain

Один детерминированный алгоритм; результат **стабилен** между пересчётами одного и того же файла. Точная спецификация — в README плагина.

**Метод v1:**

1. Взять последовательность высот всех точек трека (из `<ele>` GPX / TCX).
2. Между соседними точками вычислить Δh.
3. В сумму набора включать только положительные Δh **≥ порога** (default **3 m**) — фильтр GPS-шума.
4. Elevation loss — симметрично для отрицательных Δh ≥ порога.

Порог может быть вынесен в settings в будущем; в v1 — фиксированный 3 m.

### 4.6. Places (регионы и POI)

Place — **единая сущность** для регионов и POI. Реализуется как обычная Obsidian-заметка с геометрией во frontmatter и свободным текстом в теле. Существующая заметка может стать place-заметкой, а place-заметка может снова стать обычной заметкой после удаления специального признака.

| ID | Требование | Приоритет |
|----|------------|-----------|
| F-09 | Place = заметка с `trackdex-type: place` | Must |
| F-09h | Типы геометрии v1: `point`, `circle`, `rectangle`, `polygon` | Must |
| F-09b | Трек «в place», если **≥ 1 точка** попадает в геометрию | Must |
| F-09c | Пересечение сегмента трека с геометрией желательно учитывать, но не является обязательным для v1 | Should |
| F-09i | Создание/редактирование v1: команда «Сделать текущую заметку place» / «Редактировать geometry текущего place»; ручное правление YAML допустимо | Must |
| F-36 | POI-point: радиус `radius_m` в заметке; default — из settings плагина | Must |
| F-09a | Авто-переиндексация при изменении place (debounced) + команда вручную; если `trackdex-type: place` удалён, заметка удаляется из индекса places и треки отвязываются | Must |
| F-09j | Невалидный YAML/geometry place показывается как ошибочный place и не используется для привязки треков | Must |
| F-05f | Сортировка places по **дате последнего визита** (без порога «N дней») | Must |
| F-09f | Саджесты places / POI по координатам + онлайн-сервисы | v2+ |
| F-09g | Визуальный редактор геометрии на карте | v2+ |

В v1 `polygon` — одно внешнее кольцо без holes/multipolygon. Сложные геометрии имеют смысл вместе с визуальным редактором и относятся к backlog.

**Формат place-заметки (normative):**

```yaml
---
trackdex-type: place
trackdex-geometry:
  kind: circle          # point | circle | rectangle | polygon
  lat: 55.7558
  lon: 37.6173
  radius_m: 5000        # circle; для point — опционально (иначе default из settings)
  # rectangle: south, west, north, east
  # polygon: ring: [[lat, lon], ...]
---
```

Тело заметки — произвольный markdown пользователя (описание, заметки о месте).

## 6. Модель данных (локальный индекс, черновик)

Ниже логическая модель. Физическое хранилище не фиксируется как обязательный SQLite: допустимо любое локальное решение, которое работает на desktop/mobile и выдерживает требования к скорости и объёму.

```
tracks
  path                  TEXT PRIMARY KEY   -- vault-relative
  mtime, sha256
  status                TEXT               -- pending|indexing|indexed|stale|error
  error_message         TEXT NULL
  error_details         TEXT NULL
  title_from_file       TEXT NULL
  started_at, ended_at                  -- normalized UTC when possible
  started_at_raw, ended_at_raw          -- original file values
  timezone_source, timezone_offset       -- explicit|indexing_local|unknown
  duration_sec          REAL               -- elapsed v1
  distance_m, elevation_gain_m, elevation_loss_m
  file_metrics_json     TEXT NULL          -- values supplied by file summaries/laps if available
  computed_metrics_json TEXT NULL          -- values recalculated from points if available
  data_flags_json       TEXT               -- extensible has_* / missing_* flags
  avg_speed_mps, max_speed_mps
  sport_raw, sport_normalized
  bbox_json, polyline_simplified_json
  segments_json         TEXT NULL          -- список сегментов для multi-track, laps/intervals if cheap
  hr_avg, hr_max, power_avg, cadence_avg  NULL

places
  note_path             TEXT PRIMARY KEY
  geometry_kind         TEXT
  geometry_json         TEXT
  bbox_json             TEXT

track_places
  track_path, place_note_path
  last_visit_at         TEXT               -- max started_at треков, прошедших через place

note_track_links
  note_path, track_path
  link_text             TEXT               -- исходный wikilink

index_meta
  schema_version, last_full_scan_at, scan_paused, first_scan_approved
```

**Индекс-проходы:**

1. Треки — GPX/TCX/FIT/FIT.GZ по vault.
2. Wikilink — scan `.md` → `note_track_links`.
3. Places — заметки с `trackdex-type: place` → `places` → `track_places`.

## 7. Индексация: состояния и UX

```
pending → indexing → indexed
                  ↘ error
indexed → stale (mtime/hash изменился) → indexing
```

- До первого scan: вместо списка треков в менеджере показывается объяснение «индекс ещё не создан», что будет просканирован весь vault, поддерживаемые расширения и active exclude patterns, плюс кнопка «Начать индексирование».
- Панель внизу справа: «Indexing 42 / 380…»; не должна мешать работе.
- Sidebar: badge на непроиндексированных; фильтр «с ошибками»
- Settings: exclude patterns, units, default POI radius, пауза, ручной scan, reset index, legal/privacy info

## 8. Вне scope

### Не в v1

- Редактирование / склейка треков
- Синхронизация Strava / Garmin
- Сегменты (Strava-like) и PR по участкам
- Heatmap коллекции
- Автогеокодинг / саджесты places
- Визуальный редактор геометрии на карте
- Moving time (→ v1.1)
- Кеш тайлов (→ should, допустимо v1.1)
- Сложные дашборды и графики активности
- Детект/группировка дубликатов
- Сложные геометрии places: holes, multipolygon
- Ручные override метаданных трека: sport, date, title, hide from catalog, metric corrections
- Настройка пользовательских sport aliases/icons
- Include-фильтры сканирования и расширенные настройки scan scope
- Настройки логирования
- Отправка логов/диагностики и GitHub issue template
- Import/export настроек Trackdex
- Изменение/переинтерпретация timezone для уже проиндексированных треков
- `.gpx.gz` / `.tcx.gz`
- UI-настройка tile provider
- Исключение индекса/логов из синхронизации или рекомендации по sync после изучения Obsidian Sync/Dropbox сценариев

### v2 и далее

- Сегменты и прогресс по скорости на участках
- Heatmap
- Strava / Garmin sync (opt-in)
- Geocoding и place-саджесты
- Визуальный редактор places
- Moving time в UI и avg speed по moving time

## 9. Отличие от других плагинов

| Плагин | Trackdex |
|--------|----------|
| Map View, Leaflet | Каталог + индекс + places + stats, не inline-карты в заметках |
| Map GPX | Vault-wide local index, places, wikilink, mobile |
| Strava Sync | Offline, без облака |
| Tracker | GPS-треки, не habit tracking |

## 10. Риски

- Производительность на тысячах GPX и на mobile.
- Размер sync vault при большой коллекции.
- Конфlicts с другими map-плагинами и tile providers.
- Multi-track GPX — поведение уточняется на реальных файлах.
- Путаница с плагином **Tracker** — явно описать в README.
